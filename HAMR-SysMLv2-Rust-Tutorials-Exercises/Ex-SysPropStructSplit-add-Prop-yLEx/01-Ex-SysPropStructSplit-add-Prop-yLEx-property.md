# HAMR Exercise: Adding a System-Level Property (Part 1 - Specifying the Property)

**Purpose**: The purpose of this exercise is to get you familiar with the basic workflow for HAMR GUMBO *system-level properties*: cross-component properties of a whole composed system that are verified using only the components' GUMBO contracts (never their implementations).

Starting from the Struct Split system-property example, you will go through the steps of...
 - stating a new end-to-end property informally and convincing yourself it holds,
 - specifying the property as a GUMBO spec function and a `property` block in the system's `composition`,
 - running the HAMR type checker to check the well-formedness of your specification,
 - running HAMR code generation to produce the verification conditions (VCs) for the property,
 - reading the generated VCs to understand how the proof of the property is structured.

In Part 2 of this exercise, you will run Verus on the generated VCs, diagnose a verification failure, repair the property, and contrast two different reasons a VC can fail.

## Exercise Overview

The Struct Split example is a seven-component pipeline that manipulates a struct of two `int32` fields `x` and `y`:

- **GenStruct** (`gen`) generates a struct with arbitrary `x` and `y` values; its integration guarantee promises both fields lie in `[-100, 100]`.
- **SplitStruct** (`split_stage`) splits the struct onto two scalar ports `x0` and `y0`.
- The **x branch**: **IncX** (`incx`) computes `x1 = x0 + 1`, then **ClampX** (`clampx`) saturates the result back into `[-100, 100]` (`x2`).
- The **y branch**: **DecY** (`decy`) computes `y1 = y0 - 1`.
- **MergeStruct** (`merge_stage`) reassembles the struct (`merged.x == x2`, `merged.y == y1`) -- it carries *no* range constraints of its own.
- **ConsumeStruct** (`consume`) receives the final struct and logs it.

The example's schedule schema runs the two branches as independent `split` branches that rejoin at the label `branches_joined` before the merger.

In this exercise you will specify and verify a new **order-preservation** property:

- **REQ_SYS_YLEX**:
  If the `y` field of the struct generated by the `gen.gen` component instance is less than or
  equal to the `x` field, then the `y` field of the struct consumed by the `consume.consume`
  instance is strictly less than the `x` field.

Note the asymmetry: the antecedent is `<=` (on the generated struct) but the conclusion is strict `<` (on the consumed struct). Before specifying anything formally, you should convince yourself informally that the property is true -- and understand exactly where the strictness comes from.

* **Task:** Using the component contracts summarized above, write out (on paper) an informal argument for REQ_SYS_YLEX. Try to justify each step of the chain below, where `x`,`y` are the generated fields:

```
merged.y  ==  y - 1  <  y  <=  x  <=  x2  ==  merged.x
```

Points to check your reasoning against:

- `merged.y == y - 1` -- SplitStruct forwards `y` unchanged, DecY subtracts one, MergeStruct copies `y1` into the output struct.
- `y - 1 < y` -- arithmetic. **This single strict step is the only source of the strict `<` in the conclusion.**
- `y <= x` -- the property's antecedent.
- `x <= x2` -- this is the interesting step. IncX computes `x + 1`, but ClampX may *cancel* the increment: if `x == 100` then `x1 == 101` saturates back down to `x2 == 100 == x`. So the x branch is only guaranteed to *not decrease* `x` -- the increment cannot be credited toward strictness. Verify both cases: `x < 100` gives `x2 == x + 1 > x`, and `x == 100` gives `x2 == 100 == x`. (Why can't `x > 100` happen?)

This informal argument is worth the effort: in Part 2 you will see that the machine-checked proof decomposes into VCs that mirror it step by step -- and that one of the steps needs a fact you might not expect.

## Activity 3 - Specify the Property as a GUMBO Spec Function

Recurring predicates -- and especially *the property's own claim* -- should be named as GUMBO functions in the `functions` section of the system-level GUMBO block (the section that already defines `inRange100`, `inConsumeRange`, `inRange200`, and `mergedReflectsGen`). The function name then serves as the property's citable statement, and the concrete `property` block only has to bind it at a place.

* **Task:** In `sysmlv2/struct-split/SysPropStructSplit.sysml`, add the following function at the end of the `functions` section of the system-level GUMBO block (right after `mergedReflectsGen`):

```
        // Order preservation, named as a spec function: whenever the generated
        // struct is ordered (y <= x), the struct arriving at the consumer is
        // strictly ordered (y < x).  Strictness comes from DecY's decrement;
        // IncX's increment cannot be credited because ClampX may cancel it
        // at x == 100.
        def yStaysBelowX(m: SysPropStructSplit_Data_Model::StructXY,
                         g: SysPropStructSplit_Data_Model::StructXY): Base_Types::Boolean :=
          (g.y <= g.x) implies (m.y < m.x);
```

Note that this is a *system* specification concept: the two arguments will be bound to values observed at two different points of the schedule (the generator's output channel and the merger's output channel), something no single component's contract could express.

## Activity 4 - Add the Property Block

Now bind the claim at the place where it matters: `before consume`, the point in the schedule where ConsumeStruct is dispatched with the merged struct.

Recall from the system-properties documentation that facts do not flow between schedule places implicitly -- each VC sees only the assertion bound at its own place, so a property normally needs a chain of "carry" assertions threading facts through the schedule. The example already factors two reusable chains as abstract properties:

- `Pipeline_Ranges` carries the `[-100,100]` range facts through both branches (discharging IncX's and DecY's compute assumes along the way), and
- `Pipeline_Functional :> Pipeline_Ranges` adds value-equality carries (`x0 == gen_out.x`, `x1 == gen_out.x + 1`, the ClampX case results, `y1 == gen_out.y - 1`) on the same places.

Our property needs exactly the kind of facts `Pipeline_Functional` carries -- how the merged fields relate back to the generated fields -- so the natural move is to inherit it and state only the conclusion:

* **Task:** Add the following property at the end of the `composition nominal` block (right after `Consume_Range_Loop_Invariant`):

```
        //-----------------------------------------------------------------------------
        //  Property 4 (exercise) -- order preservation: if the generated struct has
        //  y <= x, the struct arriving at the consumer has y strictly < x:
        //  merged.y == gen.y - 1 < gen.y <= gen.x <= x2 == merged.x.
        //-----------------------------------------------------------------------------

        property Y_Stays_Strictly_Below_X :> Pipeline_Functional
          "if the generated struct has y <= x, the consumed struct has y < x" {
          before consume "the end-to-end conclusion at the consumer's dispatch":
            yStaysBelowX(merged, gen_out);
        }
```

This mirrors how the existing `End_To_End_Functional` property is written: inherit the plumbing, bind one conclusion. Whether the inherited plumbing actually carries *everything* our conclusion needs is a question only the verifier can answer -- that is Part 2.

## Activity 5 - Run HAMR Type Checking

* **Task:** Run the "HAMR Type Checking" tool from the CodeIVE command palette (or, from a terminal in `sysmlv2/struct-split/`, run `sireum hamr sysml tipe --sourcepath ../aadl-lib:. SysPropStructSplit.sysml`). You should see `Well-formed!` / 0 problems.

* **Task:** Seed an error to see what the type checker catches at the system-specification level. For example, mistype the port alias `merged` as `mergd` in your new property, or misspell the place (`before consumer`). Run "HAMR Type Checking" and observe the error. Then reset to the correct text and confirm 0 problems.

It is worth being clear about what each stage of the toolchain checks:

- **Type checking** catches syntax, name-resolution, and typing errors in the specification (including collisions between your alias/property names and SysMLv2 keywords). It does **not** tell you whether the property is *provable* -- a well-formed specification can still be unverifiable.
- **Code generation** (next activity) translates the property into verification conditions. It does not judge them either.
- **Verus** (Part 2) is the stage that decides provability.

## Activity 7 - Read the Generated Verification Conditions

Before running the verifier, take the time to read what it will be asked to prove. This is the heart of the exercise: the generated VC module is a *machine-checkable rendering of the informal proof you sketched in the Exercise Overview*.

* **Task:** Open `hamr/microkit/crates/sys_nominal_proof/src/assertions.rs` and find your function:

```rust
pub open spec fn yStaysBelowX(
  m: SysPropStructSplit_Data_Model::StructXY,
  g: SysPropStructSplit_Data_Model::StructXY) -> bool
{
  (g.y <= g.x) ==>
    (m.y < m.x)
}
```

GUMBO's `implies` became Verus's `==>`; the struct field accesses carried over directly.

* **Task:** Open `src/y_stays_strictly_below_x/assertions.rs`. There is one spec function per schedule *place*, and inheritance has been **flattened to conjunction**: every place bound by `Pipeline_Ranges` or `Pipeline_Functional` shows both parents' facts conjoined. For example, at the branch join:

```rust
/** property y_stays_strictly_below_x, bound 'at branches_joined' (place post_join_1)
  *   join: both transformed ranges are available to the merger
  */
pub open spec fn sys_assert_y_stays_strictly_below_x_post_join_1(st: SystemState) -> bool
{
  ((inRange100(st.x2) &&
    (-101i32 <= st.y1)) &&
    (st.y1 <= 99i32)) &&
    ((((st.gen_out.x < 100i32) ==>
      (st.x2 == st.gen_out.x + 1i32)) &&
      ((st.gen_out.x == 100i32) ==>
        (st.x2 == 100i32))) &&
      (st.y1 == st.gen_out.y - 1i32))
}
```

The first three conjuncts are `Pipeline_Ranges`' carries; the rest are `Pipeline_Functional`'s. Your own binding appears as:

```rust
/** property y_stays_strictly_below_x, bound 'before consume' (place after_merge_stage)
  *   the end-to-end conclusion at the consumer's dispatch
  */
pub open spec fn sys_assert_y_stays_strictly_below_x_after_merge_stage(st: SystemState) -> bool
{
  yStaysBelowX(st.merged, st.gen_out)
}
```

Note the place name: you wrote `before consume`, but the generated code uses the **canonical place name** `after_merge_stage`. Consecutive schema elements share a place -- "immediately after the merger fires" and "immediately before the consumer fires" are the same point -- and the generated code picks one canonical identifier per place regardless of which synonym you bound. Keep this in mind when navigating the VC files.

* **Task:** Open `src/y_stays_strictly_below_x/vc_sequential.rs` and skim the 17 VCs from top to bottom, matching them against the schedule schema (`gen; split_stage; split { {incx; clampx}, {decy} }; branches_joined; merge_stage; consume`). Each component transition gets a **Pre-Assert** VC (the assertion carried into the component's `before` place must imply the component's compute assumes) and a **Next-Assert** VC (the `before` assertion plus the component's contract and write frame must imply the `after` assertion). The one that will do the final work for our property is the merger's Next-Assert:

```rust
/** VC[14]: Next-Assert (task) -- post_join_1 + frames + MERGE_STAGE postcondition |- after_merge_stage */
pub proof fn vc_next_assert_task_merge_stage(pre: SystemState, post: SystemState)
  requires
    sys_assert_y_stays_strictly_below_x_post_join_1(pre),
    merge_stage_local_write_frame(pre, post),
    merge_stage_global_write_frame(pre, post),
    merge_stage::compute_spec_merge_x_guarantee(pre.x2, post.merged),
    merge_stage::compute_spec_merge_y_guarantee(pre.y1, post.merged),
  ensures
    sys_assert_y_stays_strictly_below_x_after_merge_stage(post),
{}
```

Read it as a Hoare triple: *given* the join assertion in the pre-state, *given* that MergeStruct copies `x2` and `y1` into the output struct's fields (its only contract), and *given* that everything outside MergeStruct's write set is unchanged (the write frame -- this is what lets `gen_out` in the post-state still mean the generated struct), *prove* `yStaysBelowX(merged, gen_out)`. The body is empty: Verus discharges `requires ==> ensures` as an SMT query.

* **Task:** Find the consumer's Pre-Assert VC (`vc_pre_assert_consume`, VC[15]) and note that its conclusion is `true`. This is **contract projection**: our property binds `before consume` but not `after consume`, so ConsumeStruct's contract is not used -- and, in exchange, its defensive `[-200, 200]` compute assume imposes no obligation on us. (Compare with the generated module for `consume_range_loop_invariant`, which binds `after consume` and therefore must discharge that assume.) A property only "pays" for the components whose contracts it uses.
